## BUG-1 / BUG-2（同一根因）: スケジュールコピーが店舗・担当者を無視する

- **重大度**: 🔴 Critical（データ損失・サイレント無登録）
- **報告症状**:
  - 不具合1「複数店舗があり担当者が複数の場合、スケジュール登録が不安定」
  - 不具合2「登録したはずなのにスケジュールが登録されていない」
- **該当**: `includes/rest/class-rest-schedules.php` の `copy_schedules()`

### 根本原因
フロント（`src/admin/pages/schedule/ScheduleCopyModal.jsx`）は `store_id` / `staff_id` を正しく送信している。しかしバックエンドはそれを**コピー元の抽出にしか使わず**、既存チェック・スキップ・上書き削除を **`schedule_date`（日付）だけ**で全店舗・全担当者をまたいで実行している。

- 既存チェック: `SELECT COUNT(*) ... WHERE schedule_date = %s`（店舗/担当者の絞り込みなし）
- 上書き削除: `DELETE ... WHERE schedule_date = %s AND booked_count = 0`（同上）
- `overwritten_total += $existing` も全店舗合算で誤カウント。

### 再現条件（「2つ以上の店舗/担当者グループが同じコピー先日付に枠を持つ」とき）
- **上書きOFF でコピー**: コピー先の日付に*他の*店舗/担当者の枠が1件でもあると、その日全体が「スキップ」判定になり、意図した店舗のコピーが**1件も入らない**。UI は「N件スキップ」と出すが見落としやすい → 体感「コピーしたのに登録されない」（不具合2）。
- **上書きON でコピー**: その日の*他の*店舗/担当者の未予約枠まで `DELETE` され、再挿入されるのはコピー元の店舗/担当者だけ → **他店舗・他担当者の既存スケジュールが消える**（不具合1の「不安定」＝データ損失）。単一店舗でも担当者が2人以上いれば発生。

### 修正方針
1. `copy_schedules()` の既存チェック・上書き DELETE・カウントに、コピー元と同じ `store_id` / `staff_id` の絞り込みを**必ず含める**（＝コピーの作用範囲を「その店舗×担当者」に閉じる）。
2. 上書きは日付一括 DELETE ではなく、対象 `(store_id, staff_id, schedule_date)` かつ `booked_count = 0` の枠に限定。予約済み枠は従来どおり保護。
3. `overwritten` / `skipped` カウントを対象スコープ基準に修正。
4. 併せて BUG-4（UNIQUE 制約）を入れると、コピーの多重実行でも重複が構造的に防げる。

### 検証（logic-evaluator）
- 店舗A/B・担当者2名・重なる日付を作る隔離ゲート。
- Red 例: 「店舗Bの枠を、店舗Aの枠がある日付へ上書きコピー → 店舗Aの枠が消えない」ことを assert（修正前は落ちる）。
- Red 例: 「上書きOFF で店舗Aの枠がある日付へ店舗Bをコピー → 店舗Bの枠がちゃんと作成される」。
- 回帰: 単一店舗・単一担当者のコピーが従来どおり動く。予約済み枠が保護される。

## BUG-3: 予約通知メールが届かない

- **重大度**: 🟡 Warning（コード経路は概ね正常。到達/トランスポート要因が濃厚）
- **報告症状**: 不具合3「メールが届かない」
- **該当**: `includes/class-email.php`, `includes/class-integrations.php`, `includes/class-reservation-context.php`

### 調査結果（コード経路は機能的に正常）
フロント予約（`class-rest-public.php:996`）・管理画面の手動予約（`class-rest-reservations.php:391`）の両方で `smart_booking_reservation_received` が発火し `send_receipt()` が呼ばれる。テンプレート初期値も有効化時に投入済み。したがって主因はコードではなく**送信・到達レベル**の可能性が高い。候補を切り分ける。

### 候補原因と再現条件
1. **トランスポート未設定（最有力）**: `wp_mail()` は PHP `mail()`／MTA 依存。MTA/SMTP 未設定のサーバ（新規 VPS 等）では**無言で捨てられる**。しかも `class-email.php` は `wp_mail()` の戻り値（false）を**握り潰し、ログも残さない**設計 → 失敗が見えない。
2. **到達性（Gmail 等）**: 差出人は既定 `admin_email`。これが `@gmail.com` や独自ドメインで、サーバIPから直送すると SPF/DKIM/DMARC 不整合で**拒否・迷惑メール送り**。
3. **条件付きの管理者通知**: `smart_booking_mail_admin_notify_enabled` が OFF かつ店舗メール未設定だと管理者宛は無言で送信されない。件名か本文テンプレートが空でも無言スキップ。
4. **宛先バリデーション**: `store_email` / `staff_email` が `is_email()` を通らない値だと除外される。

### 修正・対応方針
- **コード側（backend-generator）**: 送信失敗の可視化。`error_log` は審査 NG のため、`wp_mail_failed` フックで直近の失敗を transient/option に記録し、設定画面「メール通知」タブに「直近の送信に失敗しました」の注意を出す案（審査安全な範囲で）。診断性が上がり、以後の「届かない」を切り分け可能にする。
- **環境側（人間対応）**: SMTP 送信の導入（WP Mail SMTP 等）＋ 差出人ドメインの SPF/DKIM/DMARC 整備。ローカル検証では MailHog 等で To/Cc/Subject/本文を確認する。
- logic-evaluator は「`wp_mail` が正しい引数で呼ばれるか」までをゲート対象とし、**実到達性は環境要因として分離報告**する。

### 検証（logic-evaluator）
- wp-env で予約作成 → `wp_mail` の呼び出し引数（To/Cc/Subject/本文の変数展開）を検証。
- 管理者通知トグル ON/OFF・店舗メール有無・担当者CC の分岐を網羅。

## （継続）BUG-A: REST パーマリンク依存（v0.2.3 で併せて対応推奨）

- **重大度**: 🔴 高（初回利用者が高確率で遭遇）／引き継ぎドキュメント記載の既知バグ
- **症状**: パーマリンク=「基本（Plain）」のとき、React が使うハードコード `/wp-json/...` が 404 相当になり、管理画面スケジュール等でルート不一致エラー。**フロント予約フォームも同依存なら Plain 環境で予約自体が動かない可能性**。
- **不具合2との関係**: Plain パーマリンク環境では POST /schedules が 404 → 保存失敗。SchedulePage はエラートーストを出すため「完全にサイレント」ではないが、環境によっては「登録できていない」の別経路になり得る。logic-evaluator は再現時にこの条件も切り分ける。
- **修正方針**: REST URL を `esc_url_raw(rest_url())` ベースに統一（admin・frontend 両系統）。`grep -rn "wp-json" src/ includes/` で洗い出し。

## （継続）BUG-B: 管理画面ロゴが配布ZIPに未同梱

- **重大度**: 🔵 中（見た目のみ）
- **症状**: `src/admin/App.jsx` が `docs/images/SmartBookingLogo.svg` を参照。`docs/` は配布ZIP除外 → 画像割れ。
- **修正方針**: ロゴを配布対象（`build/images/` 推奨）へ移動し参照を修正。ZIP同梱を検証。

## BUG-4（構造対策）: schedules テーブルに UNIQUE 制約がない

- **重大度**: 🟡 Warning（不具合1の「不安定」を底上げする構造要因）
- **該当**: `includes/class-activator.php` の `smart_booking_schedules`（`idx_store_staff_date` は非ユニーク KEY のみ）
- **内容**: `(store_id, staff_id, schedule_date, start_time)` に UNIQUE 制約がなく、重複防止がアプリ層の read-then-write に依存。連打・並行送信で重複行ができ得る。
- **修正方針**: dbDelta で UNIQUE 制約を追加（＋ `smart_booking_db_version` bump ＋ 追加前の既存重複解消手順）。`create_item` / `copy_schedules` は制約違反時のハンドリングを持たせる。BUG-1/2 と同時に入れると効果的。

### 根本原因（実測確定）
Plain パーマリンクでは `rest_url('smart-booking/v1/')` が `http://<host>/index.php?rest_route=/smart-booking/v1/`（クエリ文字列形）を返す。両 api.js の `buildUrl()` が末尾で `return qs ? base + '?' + qs : base;` と**無条件に `?` を付与**するため、base に既に `?rest_route=` がある Plain 環境で**二重 `?`**（例 `...?rest_route=/smart-booking/v1/schedules?store_id=1`）となり `rest_route` 値が壊れて **404 `rest_no_route`**。パラメータ無し呼び出しは通るため「一部だけ失敗＝不安定」。pretty では正常。PHP（localize）は既に正しく、フロントのみの不具合。

### 変更（未コミット・未リリース・公開契約非破壊）
- `src/admin/api.js` / `src/frontend/api.js` の `buildUrl()`: `const sep = base.indexOf('?')===-1 ? '?' : '&'; return base + sep + qs;`（既存クエリがあれば `&`、無ければ `?`）。ハードコード `'/wp-json/smart-booking/v1/'` フォールバックを除去（`globalCtx.restUrl || ''` ＋不在時 console.warn）。`npm run build` エラー0で `build/*` 反映。PHP・REST 契約・`SMART_BOOKING_VERSION` 不変。
