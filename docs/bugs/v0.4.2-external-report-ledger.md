# v0.4.2 外部ユーザー報告 2件 — 調査正本(2026-07-16 起票)

調査実施: 2026-07-15(v0.4.1・main 上で読み取り専用調査・wp-env 実測済み)
方針決定: 人間確認済み(報告1=案A+関連改善 / 報告2=コード修正なし・FAQ追記のみ)

## 報告1: 「フィールドキーの部分がおかしい」

### 原文(外部Web制作者)
「最新版の予約フォームに更新して、資料送付の項目を追加して、
テスト送信してみたのですが、フィールドキーの部分がおかしいようです」

### 再現(wp-env 実測済み)
- sanitize_key('資料送付') = ''(日本語はキー化できず空)
- 空 field_key → サーバ自動採番 'field_4'(field_N)
- ユーザーがメールテンプレに {field_4} を記入 → 展開されず
  生のまま送信される(★症状の本体)
- template_vars() の展開対象は固定8変数のみ:
  {customer_name} {customer_email} {customer_phone} {reservation_id}
  {schedule_date} {schedule_time} {store_name} {staff_name}

### 生キー露出の全経路調査結果
- フロント(FormInput.jsx:195)・確認画面(ConfirmPage.jsx:220)・
  予約詳細(ReservationDetailModal.jsx:265)・CSVヘッダ
  (class-rest-reservations.php:600-603)= すべて field_label 使用で正常
- 露出はメール本文のみ(未展開の {field_N})

### 根本原因
- CustomFieldModal.jsx:301 の説明文「メール本文などで {キー名} として
  利用できます」が、実装(class-reservation-context.php:142-172 =
  固定8変数のみ)に存在しない機能を約束している(UI説明と実装の齟齬)
- 仕様 docs/smart-booking-spec.md §5.5 のテンプレート変数一覧も8個
  固定であり、実装は仕様どおり。齟齬は UI 文言側
- 副次: 日本語ラベルではキー候補が空になり(CustomFieldModal.jsx:44-55
  suggestKey)、field_key 必須(:210-214)のためユーザーが英字キーを
  手入力させられる。空送信時はサーバ採番
  (class-rest-custom-fields.php:301-307)

### 決定方針: 案A(メール変数としてカスタムフィールドを実装)
- template_vars() に custom_field_defs + reservation_meta から
  {field_key}→回答値 の変数を追加生成
- address: {key}_zip / {key}_address / {key}(結合)の3変数
- checkbox 複数値: 「、」結合
- ③条件で非表示(meta なし): 空文字展開(§8.3 の既存原則)
- 固定8変数と field_key の衝突: 固定8変数優先+field_key
  バリデーションで予約語8つを禁止する防御を追加
- ②複数フォーム: 予約の form_id のフィールド定義で展開
- 管理UI: TemplateVariableHelper にカスタム変数を動的表示+
  クリック挿入、CustomFieldModal:301 の文言を実装に一致させる、
  CustomFieldList に「メール変数 {key} として使用可」ヒント
- 関連改善: suggestKey 空時にブロックせず自動補完(サーバ採番委譲
  またはクライアント field_<連番> プリセット)
- マイグレーション不要(meta は保存済みのため遡及して展開される)
- 却下した代替案B(文言削除のみ): 生キー露出は止まるが、カスタム
  項目の回答をメールで受け取りたいという報告者の動機に応えられず、
  既に {field_N} を書いた既存テンプレートも壊れたままのため

## 報告2: 「管理者宛にメールが届かない(自動返信は届く)」

### 原文
「今回も自動返信は送られてくるのですが、管理者宛にメールが
届いていないようです」

### 判定: コードは正常(wp-env 実測で wp_mail 2回呼び出しを捕捉)
- call#1 ユーザー宛 / call#2 管理者宛(To: admin_email+店舗メール、
  Cc: 担当者)— class-email.php:96-139
- 送信マトリクスは仕様 §5.5 と整合(受付=両方、承認=ユーザーのみ、
  手動予約も受付イベント経由 class-rest-reservations.php:400)
- 未達原因はサーバ/受信側の配信性(SPF/DKIM/DMARC・迷惑メール判定)
  の可能性が高い。「ユーザー宛は届き管理者宛だけ届かない」非対称は
  配信性問題の典型(From=管理者と同一ドメインは詐称判定されやすい)

### 留意点(コードの既知の限界・修正対象外)
- v0.2.3 の送信失敗可視化(class-email.php:229-235)は wp_mail 失敗
  のみ記録。MTA 受理後の受信側拒否は検知不可
  =「バナーなし≠届いた」
- 「管理者へのメール」OFF かつ店舗メール未設定時は無記録で早期
  return(:116-118, :128-130)。既定構成では非発火

### 決定方針: コード修正なし。readme FAQ に
「送信失敗バナーが出ていなくても受信側で迷惑メール判定・拒否され
得る」旨+WP Mail SMTP 推奨を追記(0.4.2 に同乗)。
環境切り分けは制作者へ質問リストで対応済み(チャット記録)
