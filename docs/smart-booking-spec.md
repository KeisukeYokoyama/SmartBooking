## Round 1: コンセプトとスコープ

### 1.1 プラグイン基本情報

| 項目 | 決定事項 |
|------|----------|
| プラグイン名（スラッグ） | `smart-booking` |
| 表示名 | Smart Booking |
| 説明文 | 無料で多機能。最短5分で導入できるWordPress予約プラグイン |
| ライセンス | GPL v2 or later |
| 料金 | 完全無料（Pro版なし） |
| 配布先 | WordPress.org 公式ディレクトリ |

### 1.2 ビジネスモデル

- プラグイン自体は完全無料で配布する
- Pro版・有料アドオンは作らない
- プラグイン経由で「機能リクエスト・カスタマイズ相談」を受け付け、個別開発案件として収益化する
- プラグインの実績をWeb制作・コンサル受託の信頼材料として活用する

### 1.3 ターゲット業種（優先順位）

1. **弁護士・士業の相談予約** — 法律相談、税務相談、行政書士の面談予約
2. **結婚相談所** — カウンセラーとの面談予約
3. **整体院・治療院** — 施術の予約
4. **学習塾・予備校** — 体験授業、面談の予約

共通する特徴: 「人（担当者）に紐づく面談・相談型の予約」

### 1.4 多言語対応

- **日本語のみ**で開発する
- i18n（国際化）対応は初期リリースでは行わない
- .potファイルの生成は将来の拡張として扱う

### 1.5 開発依頼への導線

- **設定画面内**に「サポート・カスタマイズ相談」セクションを設置する
- 自社サイトのお問い合わせフォームへのリンクを掲載
- 過度な宣伝は避け、控えめで自然な導線にする
- WordPress.orgのガイドラインに違反しない範囲で設置する

### 1.6 競合との差別化ポイント

| Smart Booking | 競合（Booking Package等） |
|---------------|---------------------------|
| 完全無料・制限なし | 無料版に制限あり |
| 日本語ネイティブ | 多言語対応（日本語は翻訳） |
| 5分で導入可能 | 設定がやや複雑 |
| 日本の商習慣に最適化（確認画面→完了画面） | 海外の予約フローベース |
| 面談・相談型予約に特化 | 汎用的 |

### 1.7 設計原則

- **YAGNI** — 今必要ない機能は作らない
- **シンプルさ優先** — バグの温床になる複雑な仕様は避ける
- **WordPress審査基準準拠** — CDN不使用、PHPセッション不使用、外部通信なし
- **完全無料** — 機能制限、アップセル、ライセンス認証は一切なし

いずれもデフォルトではOFF。ユーザーが設定画面でAPIキー等を入力し、有効化した場合のみ通信が発生する。readme.txtに通信先・目的・タイミングを明記する。

## Round 3: 予約フロー設計

### 3.1 予約フロー全体像

エンドユーザー（予約する人）が体験する画面遷移。同一ページ内でAjaxによるステップ切替を行い、確認画面・完了画面のみ別ページに遷移する。

```
[店舗選択] → [担当者選択] → [日付選択] → [時間選択] → [フォーム入力] → [確認画面] → [完了画面]
```

#### スキップルール

| 条件 | 動作 |
|------|------|
| 店舗が1つのみ | 店舗選択ステップをスキップ |
| 担当者が1人のみ | 担当者選択ステップをスキップ |
| 店舗1つ＋担当者1人 | いきなり日付選択から開始 |

### 3.2 画面遷移方式

| 区間 | 方式 | 理由 |
|------|------|------|
| 店舗→担当者→日付→時間→フォーム | 同一ページ内Ajax切替 | スムーズなUX、ページ再読み込み不要 |
| フォーム → 確認画面 | 別ページに遷移 | 日本の商習慣に沿ったフロー |
| 確認画面 → 完了画面 | 別ページに遷移 | 予約確定の明確な区切り |

### 3.3 フォーム入力と日付・時間の表示順序

管理者が設定画面で順序を選択可能（2パターン）:

- **パターンA（デフォルト）**: 日付・時間 → フォーム入力
- **パターンB**: フォーム入力 → 日付・時間

※店舗選択・担当者選択は常にフローの先頭に位置する。順序切替の対象外。

### 3.4 カレンダーUI仕様

#### 表示モード

管理者が設定画面で選択可能な3モード:

| モード | 説明 |
|--------|------|
| 日表示のみ | 横スクロールで日付を選択（デフォルト表示日数: 7日） |
| 月表示のみ | カレンダーグリッドで日付を選択 |
| 日表示＋月表示切替 | ユーザーが日/月をトグルボタンで切替可能 |

#### 日付選択後の時間枠表示

日付を選択すると、**カレンダーの下に**空き時間枠がボタン形式で表示される。ユーザーがタップ/クリックして選択する。

#### 空き状況の表示

| 状態 | 表示 |
|------|------|
| 空きあり | 選択可能（通常色） |
| 残りわずか | 選択可能（警告色） |
| 満席 | 選択不可（グレーアウト） |
| 締切済み | 選択不可（グレーアウト） |

### 3.5 フォーム初期フィールド

プラグイン有効化時にデフォルトで設定される入力項目:

| フィールド | 種別 | 必須 | 備考 |
|-----------|------|------|------|
| 氏名 | テキスト | ✅ 必須 | フルネーム |
| メールアドレス | email | ✅ 必須 | 確認メール送信先 |
| 電話番号 | tel | ✅ 必須 | 緊急連絡先 |

管理者はカスタムフィールド機能で項目を自由に追加・編集・削除可能。

### 3.6 確認画面

| 項目 | 仕様 |
|------|------|
| 表示内容 | 選択した店舗・担当者・日時・入力情報の一覧 |
| 操作 | 「予約を確定する」ボタン / 「修正する」ボタン |
| 修正ボタン押下時 | 入力画面に戻る（入力内容は保持） |
| 遷移方式 | 別ページ |

### 3.7 完了画面

| 項目 | 仕様 |
|------|------|
| 表示内容 | 完了メッセージ、予約番号、予約日時 |
| 完了メッセージ | 管理者が設定画面でカスタマイズ可能 |
| 遷移方式 | 別ページ |

### 3.8 予約締切設定

管理者が設定画面で以下のいずれか（または両方）を設定可能:

| 設定 | 例 |
|------|-----|
| ○日前まで | 「3日前まで」→ 4/27の予約は4/24まで受付 |
| ○時間前まで | 「2時間前まで」→ 14:00の予約は12:00まで受付 |

両方設定された場合は、より厳しい方（早い締切）が適用される。

### 3.9 表示期間設定

フロントエンドの予約フォームで、何日先まで予約可能な日付を表示するかを管理者が設定可能:

| 設定 | 例 |
|------|-----|
| ○日先まで | 「7」→ 今日から7日先まで表示（デフォルト） |

例: 7と入力すると今日から7日後までの日付が表示される。30なら1ヶ月先まで。

## Round 4: 管理画面設計

### 4.1 メニュー構成

WordPress管理画面のサイドバーにトップレベルメニューとして配置。

```
Smart Booking
├── スケジュール      ← トップページ（メニュークリック時に表示）
├── 予約一覧
├── 店舗・担当者
├── フォーム設定
└── 設定
```

### 4.2 スケジュール管理画面

| 項目 | 仕様 |
|------|------|
| UI形式 | カレンダー形式 |
| 操作 | 日付をクリックして時間枠を編集 |
| 表示対象 | 店舗・担当者ごとにフィルタリング可能 |
| トップページ | Smart Bookingメニュークリック時に最初に表示される画面 |

#### スケジュール追加

モーダルダイアログで以下を入力:

| 項目 | 仕様 |
|------|------|
| 日付 | 日付ピッカーで選択（必須） |
| 時間枠設定 | チェックボックスで有効/無効 |
| 時間枠単位 | セレクト（30分 / 60分 / 90分 / 120分） |
| 時間枠 | 開始時間 + 予約可能数の組み合わせを複数追加可能 |
| 時間枠追加 | 「時間枠を追加」ボタンで行を追加 |
| 利用可能 | チェックボックス（臨時休業等に使用） |

#### スケジュールコピー

既存のスケジュールをコピー元として、他の日付に複製する機能。モーダルダイアログで操作。

**コピー元情報の表示**: 日付、時間枠（在庫数含む）、時間枠単位、利用可能状態を表示。

**コピー先の選択方式（2モード）**:

| モード | 操作 |
|--------|------|
| 日付を個別選択 | 日付ピッカーで1日ずつ追加。「日付を追加」ボタンで複数日指定可能 |
| パターンで選択（毎週○曜日） | 曜日チェックボックス（日〜土）で対象曜日を選択し、期間（開始日〜終了日）を指定 |

**上書きオプション**: 「既存のスケジュールがある日付も上書きする」チェックボックス。チェックを外すと、既存スケジュールがある日付はスキップされる。

### 4.3 予約一覧画面

| 項目 | 仕様 |
|------|------|
| UI形式 | リスト表示（WordPress投稿一覧風） |
| 表示項目 | 予約番号、予約者名、日時、店舗、担当者、ステータス |
| 操作 | ステータス変更（承認・キャンセル）、詳細表示、手動予約作成 |
| フィルタ | ステータス別、日付範囲、店舗・担当者別 |
| CSV | 一覧をCSVでエクスポート可能 |

### 4.4 店舗・担当者管理画面

| 項目 | 仕様 |
|------|------|
| 店舗管理 | 店舗の追加・編集・削除・有効/無効切替 |
| 担当者管理 | 担当者の追加・編集・削除・所属店舗の設定 |
| 表示情報 | 店舗名、担当者名、所属店舗 |

### 4.5 フォーム設定画面

| 項目 | 仕様 |
|------|------|
| 機能 | カスタムフィールドの追加・編集・削除・並び替え |
| フィールド種別 | テキスト、メール、電話番号、テキストエリア、セレクトボックス、ラジオボタン、チェックボックス |

### 4.6 設定画面

タブ切替形式で以下の5タブを配置:

| タブ | 内容 |

| メール通知 | メールテンプレート編集、送信先設定、差出人設定 |
| 外部連携 | Googleカレンダー連携設定、ChatWork通知設定 |
| デザイン | カラーカスタマイズ（ボタン色、日付選択色、時間帯選択色、必須マーク色、フォーカス色） |
| サポート | 使い方ガイド、よくある質問、カスタマイズ相談への導線 |

### 4.7 リード獲得導線

| 項目 | 仕様 |
|------|------|
| 配置場所 | 設定画面の「サポート」タブ内のみ |

### 5.2 DB設計

6テーブル構成。プレフィックスは `{$wpdb->prefix}smart_booking_`。

### 5.4 WordPress審査対策

旧コードの却下原因を踏まえ、以下を必須ルールとする:

| 項目 | ルール |
|------|--------|
| テーブル作成 | `register_activation_hook` でのみ実行。`init` での毎回実行は禁止 |
| セッション | PHPネイティブセッション不使用。状態管理はReact Stateで行う |
| 外部CDN | 不使用。全ライブラリを `@wordpress/scripts` でバンドル同梱 |
| 認証・認可 | 全REST APIエンドポイントにnonce検証 + `current_user_can` 権限チェック |
| SQLインジェクション対策 | 全DBクエリに `$wpdb->prepare()` 使用 |
| XSS対策 | 全出力に `esc_html()` / `esc_attr()` / `wp_kses_post()` 適用 |
| 外部通信 | Google Calendar / ChatWorkはデフォルトOFF。readme.txtに通信先・目的・タイミングを明記 |
| デバッグコード | `error_log()` は本番コードに含めない |
| ファイル操作 | `file_get_contents()` 不使用。必要な場合は `WP_Filesystem` を使用 |
| 直接アクセス防止 | 全PHPファイル冒頭に `if (!defined('ABSPATH')) exit;` |

#### 接頭辞の方針（設計判断の記録）

WordPress.org審査フィードバックを受け、名前空間衝突を回避するため、DBテーブル名・option/transientキーの接頭辞を `smb_` から `smabo_` へ統一した（テーブル: `{$wpdb->prefix}smabo_*`、オプション: `smabo_*`、transient: `smabo_gcal_token`）。

ただし、REST APIのエラーコード（例: `smb_reservation_full` / `smb_staff_not_found`、52種）は接頭辞 `smb_` を維持する。理由は以下のとおり:

- エラーコードは名前空間ではなく**識別子**であり、DBやWordPressのグローバル名前空間に登録されないため衝突リスクがない（審査対象外）。
- 内部完結（消費するのは当プラグインのReactのみ。本仕様書・外部API契約に未記載、外部連携も非依存）で、唯一のJS側コード分岐も `err.status` フォールバックを併用しているため、リネームの技術的必然がない。
- よって安定性を優先し据え置く。**`smb_` のエラーコードは移行漏れではなく意図的な設計判断であり、一括置換しないこと。**

> 注: 本仕様書 5.2 / 5.11 等のテーブル名・オプション名の表記は、サブ4-5c-2 で `smabo_` に統一済み（コード上も `smabo_` へ更新済み）。
>
> なお、`docs/spec-amendment-*` および `docs/legacy-ui-handover/` 配下の歴史的改訂文書は、仕様変更の履歴を記録する目的の文書であり、当時の表記（`smb_*`）のまま据え置いている。現行の表記とは異なる場合がある。

##### 追記（段階8 / 2026-06-24）: `smabo_` → `smart_booking_` への再統一

> 上記「`smb_` → `smabo_`」（段階4）の記録は当時の判断として保持するが、**現行コードでは下記のとおり `smart_booking_` に再統一済み**である。

WordPress.org 再審査（R smart-booking/liberdadeinc/20Jun26/T2、2026-06-23）で、主プレフィックス `smart_booking_`（クラス・関数・フック・REST namespace）と段階4 で導入した DB 系プレフィックス `smabo_`（テーブル名・option・transient・メタキー）の**混在**を再指摘された。Plugin Check は 0/0 で通過していたが、審査者は「プラグイン内で複数プレフィックスが混在している」ことを問題視した判断。

これを受け、DB 系プレフィックスを主プレフィックス `smart_booking_` に統一した（テーブル: `{$wpdb->prefix}smart_booking_*`、オプション: `smart_booking_*`、transient: `smart_booking_gcal_token`、メタキー: `_smart_booking_gcal_event_id`）。

- 既存ユーザーはゼロ（WordPress.org 公開前）のため、DB マイグレーションは実装せず、**純粋リネーム方式**を採用した。有効化時に新プレフィックスのテーブル・オプションを再シードする。
- エラーコード `smb_*`（52種）は引き続き維持する。理由は上記（段階4）の判断と同一で、識別子でありプレフィックス規約の対象外のため。

### 5.5 メール通知仕様

#### 送信タイミング

| タイミング | ユーザーへ | 管理者へ |
|-----------|-----------|---------|
| 予約受付時（ユーザーが送信） | ✅ 「予約を受け付けました」 | ✅ 「新しい予約が入りました」 |
| 予約承認時（管理者が承認） | ✅ 「予約が確定しました」 | — |

※ユーザー側のキャンセル機能はv1では実装しない。キャンセルは電話・メールで連絡を受け、管理者が管理画面でステータスを変更する運用。

#### 送信先

- **ユーザー宛**: 予約時に入力されたメールアドレス（`smart_booking_reservations.customer_email`）
- **管理者宛**: 店舗のメールアドレス（`smart_booking_stores.email`）。担当者メール（`smart_booking_staff.email`）が設定されている場合はCCとして送信

#### テンプレート変数

管理者がメール本文に挿入可能な変数:

| 変数 | 内容 |
|------|------|
| `{customer_name}` | 予約者氏名 |
| `{customer_email}` | メールアドレス |
| `{customer_phone}` | 電話番号 |
| `{reservation_id}` | 予約番号 |
| `{schedule_date}` | 予約日（例: 2026年5月1日（金）） |
| `{schedule_time}` | 予約時間（例: 14:00〜） |
| `{store_name}` | 店舗名 |
| `{staff_name}` | 担当者名 |

#### メール設定項目（設定画面「メール通知」タブ）

- 差出人名
- 差出人メールアドレス
- 予約受付メール: 件名・本文（ユーザー宛 / 管理者宛）
- 予約承認メール: 件名・本文（ユーザー宛）

### 5.8 同時予約の競合防止

予約確定時にアトミックな1クエリで空き枠を確認・更新する:

```sql
UPDATE smart_booking_schedules
SET booked_count = booked_count + 1
WHERE id = %d AND booked_count < capacity
```

影響行数が0の場合は「この時間枠は満席になりました」とエラーを返す。トランザクションは使わず、この1クエリで競合を防止する。
