# Smart Booking 仕様修正: システムエンティティ方式（店舗・担当者の自動管理）

## 変更の背景

プラグイン有効化時にデフォルトの店舗・担当者が自動作成される仕様について、以下の問題がある:

1. ユーザーが「なぜ勝手に店舗・担当者が作られるのか」と不安になる
2. 店舗の概念がないビジネス（個人弁護士など）では不自然
3. 担当者の概念がないビジネス（結婚相談所など）では不自然
4. デフォルトデータを無効化するとスケジュール管理画面がブロックされる

## 設計方針

DB の NOT NULL 制約（store_id, staff_id）を維持しつつ、ユーザーには見えない「システムエンティティ」を裏に持つ。ユーザーは店舗・担当者を一切意識せずにスケジュール作成・予約運用ができる。

---

## 変更箇所

### 変更1: DBスキーマ — is_system カラム追加

smb_stores テーブル:

| カラム | 型 | 説明 |
|--------|-----|------|
| is_system | tinyint(1) DEFAULT 0 | 1=システム自動作成エンティティ（ユーザーに非表示） |

smb_staff テーブル:

| カラム | 型 | 説明 |
|--------|-----|------|
| is_system | tinyint(1) DEFAULT 0 | 1=システム自動作成エンティティ（ユーザーに非表示） |

### 変更2: DBマイグレーション

既存環境（既にプラグインが有効化済み）への対応:

- class-activator.php の dbDelta で is_system カラムを追加
- 既存のデフォルト店舗・担当者（id=1 で name が初期値のもの）に is_system=1 を設定する UPDATE を実行
- smb_db_version を上げてマイグレーション管理

新規インストール環境:
- デフォルト店舗・担当者を is_system=1 で INSERT（従来通りだが is_system フラグ付き）

### 変更3: プラグイン有効化時のデフォルトデータ

| エンティティ | name | is_system | is_active | 備考 |
|-------------|------|-----------|-----------|------|
| デフォルト店舗 | デフォルト | 1 | 1 | ユーザーに見えない |
| デフォルト担当者 | デフォルト | 1 | 1 | ユーザーに見えない、store_id=デフォルト店舗 |

### 変更4: 管理画面 — 店舗・担当者一覧

- is_system=1 のエントリは一覧に**表示しない**
- ユーザーには「まだ店舗が登録されていません」のように見える
- is_system=1 のエントリは削除不可（REST APIで400を返す）

### 変更5: 管理画面 — スケジュール管理

| 状態 | ドロップダウンの挙動 |
|------|-------------------|
| ユーザー作成の店舗・担当者がない | システムエンティティを自動使用。ドロップダウン自体を非表示にする |
| ユーザー作成の店舗がある、担当者がない | 店舗ドロップダウンはユーザー作成分のみ表示。担当者はシステム担当者を自動使用（非表示） |
| ユーザー作成の店舗・担当者がある | 両方ともユーザー作成分のみ表示。システムエンティティは選択肢に出さない |

スケジュール作成時の自動紐づけ:
- 店舗ドロップダウンが非表示 → システム店舗の id を自動セット
- 担当者ドロップダウンが非表示 → システム担当者の id を自動セット

### 変更6: 管理画面 — 予約一覧

- is_system=1 の店舗・担当者に紐づく予約は、店舗名・担当者名カラムを「-」で表示
- フィルタの店舗・担当者ドロップダウンにはシステムエンティティを含めない
- CSVエクスポートでも同様に「-」で出力

### 変更7: フロント予約フォーム

- is_system=1 の店舗・担当者は、既存の show_store_front / show_staff_front の設定に関わらず常にスキップ
- スキップルール（優先度順）:
  1. is_system=1 のエンティティしかない → 必ずスキップ
  2. show_store_front=OFF / show_staff_front=OFF → スキップ
  3. ユーザー作成のエンティティが1つのみ → スキップ
  4. ユーザー作成のエンティティが複数 → 選択画面を表示

### 変更8: 確認画面・完了画面

- is_system=1 の店舗に紐づく予約 → 店舗名を表示しない
- is_system=1 の担当者に紐づく予約 → 担当者名を表示しない
- show_store_front / show_staff_front の既存ロジックと組み合わせる

### 変更9: メール通知

- ユーザー宛メール: is_system=1 の店舗名・担当者名は出力しない（既存の show_*_front ロジックに加えて）
- 管理者宛メール: is_system=1 の場合、店舗名「デフォルト」、担当者名「デフォルト」をそのまま出力（管理者は内部構造を理解しているため）

### 変更10: 外部連携（Google Calendar・ChatWork）

- Google Calendar イベント: is_system=1 の場合、イベントタイトルに店舗名・担当者名を含めない（例: 「予約: 山田太郎 14:00〜」）
- ChatWork 通知: 同上

### 変更11: REST API

**店舗API（/smart-booking/v1/stores）:**
- GET（一覧）: is_system=1 を除外して返す
- GET（単体）: is_system=1 でも返す（内部参照用）
- DELETE: is_system=1 は削除不可（400 Bad Request）
- POST/PUT: is_system フィールドは変更不可（リクエストに含まれても無視）

**担当者API（/smart-booking/v1/staff）:**
- 同上

**スケジュールAPI:**
- POST: store_id / staff_id が未指定の場合、システムエンティティを自動補完
- GET（管理画面向け）: 変更なし（is_system=1 のスケジュールも返す）
- GET（フロント公開API）: 変更なし（show_*_front ロジックで制御済み）

**設定API:**
- 変更なし

### 変更12: 既存テストへの影響

- 既存テストの多くは id=1 のデフォルト店舗・担当者を使用している
- is_system=1 になることで、REST API の GET 一覧から除外される
- テストの helpers.js / phase*-helpers.js でデータ作成時に is_system=0 の店舗・担当者を明示的に作成する必要がある
- または、テスト用にシステムエンティティとは別のテストデータを seed する

---

## 変更しないもの

- DB の NOT NULL 制約（store_id, staff_id）
- smb_schedules, smb_reservations のスキーマ
- アトミック UPDATE による競合防止ロジック
- ショートコードの store_id パラメータ（従来通り動作）
- wp_options に保存する設定値のスキーマ
