# Smart Booking 機能追加仕様書（フォーム別メール文面）

## ドキュメント情報

| 項目 | 内容 |
|------|------|
| 対象バージョン | v0.5.0 |
| ステータス | 仕様確定（2026-07-16 合意） |
| 背景 | 外部ユーザー要望「フォームによって変数（カスタムフィールド）が異なるため、フォーム別にメール文面を変えたい」 |
| 前提 | v0.4.2 リリース済み（メール変数展開・フォーム別変数ヘルパーが実装済み） |
| 設計原則 | YAGNI / シンプルさ優先 / 3原則（壊さない・デグレさせない・影響範囲を調査する） |

## 概要

メールテンプレート（受付ユーザー宛 / 受付管理者宛 / 承認ユーザー宛）を、フォームごとに上書きできるようにする。

**設計方式: グローバル既定 + フォーム別オーバーライド**

- 既存の「設定 > メール通知」は**共通（既定）テンプレート**としてそのまま維持
- フォームごと・メール種別ごとに「このフォーム専用の文面を使う」トグルを設け、ON のときだけ専用の件名・本文が使われる
- 未設定（OFF）は共通テンプレートにフォールバック
- **初期状態は全フォーム全種別 OFF = 既存ユーザーの挙動は1文字も変わらない**

## 仕様

### 上書きの粒度

| 項目 | 仕様 |
|------|------|
| 粒度 | **フォーム × メール種別**（3種それぞれ独立トグル） |
| 種別 | 予約受付（ユーザー宛）/ 予約受付（管理者宛）/ 予約承認（ユーザー宛） |
| 上書き対象 | 件名・本文のみ |
| 差出人（From名/アドレス） | **共通のまま**（フォーム別にしない。配信性の事故要因を増やさない） |
| デフォルトフォーム | 特別扱いせず、他フォームと同様に上書き可能 |

### UI: フォーム設定に「メール」タブを新設

```
フォーム設定
┌──────────────────────────────────────────────┐
│ [無料体験フォーム ▼] [編集] [+ フォームを追加] │
│ [smart_booking form_id="2"] [コピー]           │
├──────────────────────────────────────────────┤
│ フィールド設定 | メール | テーマ設定            │ ← タブ追加
├──────────────────────────────────────────────┤
│ ℹ️ 未設定の項目は「設定 > メール通知」の         │
│    共通文面が使われます                        │
│                                              │
│ ▼ 予約受付メール（ユーザー宛）                  │
│   ☑ このフォーム専用の文面を使う               │
│   件名 [ ... ]                                │
│   本文 [ ... ]                                │
│   [利用できる変数] ← 選択中フォームの変数のみ   │
│                                              │
│ ▼ 予約受付メール（管理者宛）  ☐（共通を使用中） │
│ ▼ 予約承認メール（ユーザー宛）☐（共通を使用中） │
└──────────────────────────────────────────────┘
```

| 項目 | 仕様 |
|------|------|
| タブ名 | 「メール」（設定画面の「メール通知」と区別するため短名） |
| タブの従属 | **選択中フォームに従属**（セレクタの影響を受ける。テーマ設定タブとは逆） |
| トグルON時 | **その時点の共通テンプレートを複製してプリセット**（白紙から書かせない） |
| トグルOFF時 | 専用文面は**保存したまま無効化**（再ONで復活。破棄しない） |
| OFF時の表示 | 共通文面を薄色（読み取り専用）でプレビュー表示 |
| 変数ヘルパー | **選択中フォームの変数のみ表示**（v0.4.2 のフォーム別ヘルパーを再利用）。クリック挿入対応 |
| 未定義変数 | 既存原則どおり空文字に展開 |

### 共通側（設定 > メール通知）への注記

| 項目 | 仕様 |
|------|------|
| 注記表示 | 共通テンプレートの各セクションに、専用文面を使用中のフォームがあれば注記する：「※ ○○フォームは専用文面を使用中（このテンプレートの変更は反映されません）」 |
| リンク | 注記からフォーム設定のメールタブへ遷移できる |
| 目的 | 「共通を直したのに反映されない」という混乱の防止 |

### データ設計

| 項目 | 仕様 |
|------|------|
| 保存先 | `smart_booking_forms` に `mail_overrides` カラム追加（LONGTEXT / JSON、NULL = 全種別で共通使用） |
| JSON構造 | `{"reception_user": {"enabled": true, "subject": "...", "body": "..."}, "reception_admin": {...}, "approval_user": {...}}` |
| option方式を不採用の理由 | 最大10フォーム×3種×2 = 60キーの乱立を回避。フォーム削除時にカラムごと自然消滅する |
| マイグレーション | v0.5.0 ゲート（SMART_BOOKING_VERSION 基準・冪等）でカラム追加のみ。既存データ変換なし |

### 送信時の解決ロジック

| 項目 | 仕様 |
|------|------|
| 判定箇所 | class-email.php の送信時に1箇所で判定：予約の form_id → フォームの mail_overrides → 該当種別が enabled なら専用、なければ共通 option |
| 削除済みフォームの予約 | フォーム削除で mail_overrides も消滅するため、自然に共通文面へフォールバック（承認メール等） |
| 手動予約 | 従来どおりメール送信なし（変更なし） |

### REST API

| 項目 | 仕様 |
|------|------|
| エンドポイント | 新設なし。既存 `PUT /forms/{id}` に `mail_overrides` を追加 |
| バリデーション | enabled が true の種別は subject/body 必須。サニタイズは既存メールテンプレートと同一基準 |

## 完了条件

- フォーム×種別ごとにトグルON/OFFでき、ONでのみ専用文面が使われる
- トグルON時に共通文面がプリセットされる。OFF→再ONで専用文面が復活する
- 専用文面のメールで、そのフォームのカスタムフィールド変数が展開される
- 変数ヘルパーに選択中フォームの変数のみ表示される
- 共通側に「専用文面使用中」の注記が表示される
- **全トグルOFF（初期状態）の送信内容が v0.4.2 と1文字も変わらない**（デグレ最重要）
- フォーム削除後の既存予約への承認メールが共通文面で送信される
- v0.4.2 → v0.5.0 マイグレーション（カラム追加）が冪等に完了する

## 不採用・保留事項（判断記録）

| 項目 | 判定 | 理由 |
|------|------|------|
| フォームごとに完全な別テンプレート一式（案1） | ❌ 不採用 | 最大30テンプレートの管理負荷・既定概念の欠如。案2で包含 |
| {form_name} 変数のみの対応（案3） | ❌ 不採用（単独では） | フォームごとに変数が異なるという要望に応えられない。ただし {form_name} 変数自体は有用なら実装時に同乗可 |
| フォーム単位の一括ON/OFF | ❌ 不採用 | 「受付だけ変えたい」に3種全部書かせることになる |
| 差出人のフォーム別設定 | ❌ 不採用 | SPF/DKIM/DMARC の事故要因増。需要薄 |
| OFF時の専用文面破棄 | ❌ 不採用 | 誤操作での文面消失を防ぐ（保存したまま無効化） |
| デフォルトフォームの特別扱い（常に共通） | ❌ 不採用 | 例外ルールを増やさない |
