# ① 店舗・担当者の呼び方設定

## 概要

「店舗」「担当者」という名称を、業種に合わせて管理者が変更できるようにする（例: サロン / 校舎、スタッフ / 先生）。

## 設計方針

- **マスターテーブルは作らない。** 設定に呼び方テキスト2つを追加するのみ
- **反映はフロントエンド（予約フォーム）のみ。** 管理画面は「店舗・担当者」のまま固定
  - 理由: 管理画面まで変えると、サポート・ドキュメント・スクリーンショットが全ユーザーで食い違い、カスタマイズ受託時の意思疎通に実害が出るため

## 仕様

| 項目 | 内容 |
|------|------|
| 設定場所 | 設定 > 基本設定タブ |
| 設定項目 | 「店舗の呼び方」「担当者の呼び方」（テキスト入力） |
| デフォルト値 | 店舗 / 担当者 |
| 空文字時 | デフォルト値にフォールバック |
| 保存先 | 個別 option 2つ（`smart_booking_store_label` / `smart_booking_staff_label`）を既存の設定 whitelist に追加 |
| フロント反映箇所 | ステップ見出し、確認画面のラベル、エラーメッセージ |
| フロントへの受け渡し | 既存の `GET /public/settings` に2キーを追加 |
| メールテンプレート | 影響なし（`{store_name}` 等は値の埋め込みのため変更不要） |

## 完了条件

- 設定画面で呼び方を変更・保存できる
- フロント予約フォームの見出し・確認画面・エラーメッセージに反映される
- 空文字保存時にデフォルト値で表示される
- 管理画面の表記が変わらないこと

# ③ 条件フィールド

## 概要

radio / select の選択値に応じて、別のフィールドを表示/非表示にする（例:「資料送付: 希望する」を選んだときだけ住所フィールドを表示）。表示中のみ必須バリデーションが有効になる。

## 設計方針（3制約）

複雑化・バグの温床（条件の組み合わせ爆発）を防ぐため、以下を仕様として固定する:

| # | 制約 | 内容 |
|---|------|------|
| 1 | 条件は1つだけ | 「フィールドXの値がYと等しいとき表示」のみ。AND/OR・複数条件なし |
| 2 | 親は radio / select のみ | text 等を親にすると「部分一致・空文字」の仕様が濁るため |
| 3 | ネスト禁止 | 条件付きフィールドを別フィールドの親にできない（1段のみ） |

## DB 変更

`smart_booking_custom_fields` に2カラム追加:

| カラム | 型 | 説明 |
|--------|-----|------|
| condition_field_key | varchar(100) NULL | 親フィールドのキー。NULL = 常時表示 |
| condition_value | varchar(255) NULL | 親フィールドがこの値のとき表示 |

## 管理画面 UI

- フォーム設定 > フィールド編集モーダルに「表示条件」セクションを追加
- 「表示条件を設定する」チェックボックス → ON で「フィールド選択（radio/select のみ列挙、自分自身と条件付きフィールドは除外）」+「値選択（親の選択肢から選ぶ）」
- 親フィールドを削除する場合、それを親とする条件付きフィールドが存在するときは削除をブロックし、依存フィールド名を提示する

## 動作仕様（重要）

| 項目 | 仕様 |
|------|------|
| 非表示フィールドの値 | **送信時に破棄する。** React 側で state から除外し、サーバー側でも条件不成立なら無視する（「希望する→住所入力→やっぱり希望しない」で古い値が残るバグを構造的に防止） |
| 必須バリデーション | **表示中のみ有効。** フロントとサーバー（`POST /public/reservations`）の両方で同一ロジックで判定 |
| サーバー側判定 | 送信された親フィールドの値から条件を再評価する。**フロントの判定結果を信用しない** |
| 確認画面 | 非表示だったフィールドは行ごと表示しない |
| CSV / 予約詳細 | 破棄された値は空欄（meta 行を作らない） |

## 完了条件

- 親の選択値に応じてフィールドが出現/消失する
- 非表示に戻したフィールドの入力値が予約データに残らない
- 表示中のみ必須エラーが発動する（フロント・サーバー両方）
- 親フィールド削除時に依存チェックが働く
- ネスト・複数条件が UI 上で設定できないこと

# ④ 住所フィールド（郵便番号自動入力）

## 概要

フィールドタイプに「住所」を追加。郵便番号を入力すると住所欄に自動補完される。

## 設計方針

- **「郵便番号」+「住所」の2入力のみの複合フィールド。** 都道府県セレクト等への分割はしない（条件フィールド・CSV との組み合わせ複雑化を回避）
- UX: 郵便番号入力 → 住所欄に「東京都渋谷区渋谷」まで自動補完 → ユーザーが番地を追記
- 住所検索 API 障害時は自動補完が効かないだけで手入力可能とする（**フェイルソフト**。エラーで送信をブロックしない）

## 仕様

| 項目 | 内容 |
|------|------|
| フィールドタイプ | `address`（field_type に追加） |
| 入力要素 | 郵便番号（7桁、ハイフン任意）+ 住所（text） |
| 保存 | `reservation_meta` に `{field_key}_zip` / `{field_key}_address` の2キー |
| CSV | 2列追加（郵便番号 / 住所） |
| 必須設定 | is_required は複合全体に適用（郵便番号・住所の両方必須） |
| 条件フィールドとの関係 | 住所フィールドは条件付き表示の**子になれる**（③のユースケースの本命）。親にはなれない（radio/select ではないため） |
| 自動入力 ON/OFF | フィールド設定に「郵便番号から住所を自動入力する」チェック（デフォルト ON） |

## 外部通信（審査対応・必須）

- 住所検索は zipcloud 等の無料 API をブラウザから直接呼び出す
- **本機能により初めて「予約フロー中に外部通信が発生しうる」ため、readme.txt の External services セクションに通信先・目的・タイミングを追記する**（承認済みプラグインでも規約は継続適用）
- 通信が発生するのは「管理者が住所フィールドを追加し、自動入力を ON にしている場合のみ」= 既存の「明示的有効化のみ通信」原則と整合
- 日本郵便データの同梱は行わない（約12MB + 更新負担）

## 完了条件

- フィールドタイプ「住所」を追加でき、フォームに郵便番号+住所が表示される
- 郵便番号入力で住所が自動補完される
- API 到達不可でも手入力で予約が完了する
- 自動入力 OFF 時に外部通信が一切発生しない
- CSV に2列が正しく出力される
- readme.txt に外部通信の開示が追記されている

# ② 複数フォーム

## 概要

「無料相談」「無料体験」など用途別に複数のフォーム定義を持てるようにする。

## 設計方針（最重要）

- **スケジュールは分けない。分けるのはフォーム定義のみ。**
  - 全フォームが同一の店舗×担当者スケジュール（空き枠）を共有する
  - 前提: 同一担当者の同一時間枠は用途を問わず取り合う（面談型予約の実態）
  - 予約枠管理・アトミック UPDATE による競合防止・締切ロジックには**一切手を入れない**
- 上限: **10フォーム**（`SMART_BOOKING_MAX_FORMS = 10` 定数でハードコード。設定画面には出さない）
  - 目的は性能ではなく UI 崩壊防止。境界テストは「10個目まで作成可 / 11個目は不可」の2点で済む

## UI 構成: セレクタ切替の1階層（一覧ページなし）

```
フォーム設定
┌─────────────────────────────────────────┐
│ [標準フォーム ▼] [編集] [+ フォームを追加] │ ← セレクタ + 名前編集 + 追加
├─────────────────────────────────────────┤
│ （フィールド設定タブ｜テーマ設定タブ）      │ ← 既存のまま
│  選択中フォームのフィールド一覧・追加・並び替え │
└─────────────────────────────────────────┘
```

| 項目 | 仕様 |
|------|------|
| セレクタ | 既存フォーム設定画面の最上部にドロップダウン。選択中フォームのフィールドを表示 |
| フォーム追加 | 「+ フォームを追加」→ モーダルで名前入力 → 作成 |
| 追加時の初期化 | **初期フィールド（氏名・メール・電話、is_system=1）を自動生成**（有効化時シード処理の再利用）。フィールドゼロの壊れたフォームを構造的に排除し、メール通知の前提（customer_name 等）を常に成立させる |
| 名前変更 | セレクタ横「編集」→ 名前編集モーダル |
| フォーム複製 | v1 では実装しない（要望が来てから検討） |
| テーマ設定タブ | **全フォーム共通**（セレクタの影響を受けない）。フォームごとのカラー分けは行わない |
| 1フォーム運用時 | セレクタはデフォルトフォームのみ表示。従来と実質同じ見た目 |

## DB 設計

### 新テーブル: smart_booking_forms

| カラム | 型 | 説明 |
|--------|-----|------|
| id | bigint UNSIGNED PK AUTO_INCREMENT | 主キー |
| name | varchar(255) NOT NULL | フォーム名 |
| is_default | tinyint(1) NOT NULL DEFAULT 0 | デフォルトフォーム（常に1件のみ） |
| sort_order | int NOT NULL DEFAULT 0 | 表示順 |
| created_at / updated_at | datetime NOT NULL | |

### 既存テーブルへの追加

| テーブル | 追加カラム | 用途 |
|----------|-----------|------|
| smart_booking_custom_fields | form_id bigint UNSIGNED NOT NULL | フィールドの所属フォーム |
| smart_booking_reservations | form_id bigint UNSIGNED NOT NULL | どのフォーム経由の予約か（一覧フィルタ・CSV 用） |

### field_key の一意制約変更（最重要・要注意）

- 現状: `field_key` はテーブル全体で UNIQUE
- 変更後: **`(form_id, field_key)` の複合 UNIQUE**（「name」等が全フォームに存在するため）
- dbDelta はインデックス変更が苦手なため、マイグレーション処理で明示的に `DROP INDEX` + `ADD UNIQUE KEY` を行う
- **v0.3.x → v0.4.0 で唯一の「既存ユーザーに影響する DB 変更」。最も慎重にテストする箇所**

## マイグレーション（既存ユーザー向け・冪等・activator のバージョン判定で1回のみ実行）

1. `smart_booking_forms` 作成、「標準フォーム」（is_default=1）を1件シード
2. 既存 `custom_fields` 全行に `form_id = デフォルトフォームの id` を付与
3. 既存 `reservations` 全行にも同様
4. field_key の UNIQUE 制約を複合キーに張り替え

## REST API

### 管理エンドポイント（すべて manage_options 必須）

| メソッド | パス | 用途 |
|---------|------|------|
| GET | /admin/forms | フォーム一覧 |
| POST | /admin/forms | フォーム作成（上限10チェック、初期フィールド自動生成） |
| PUT | /admin/forms/{id} | 名前変更 |
| DELETE | /admin/forms/{id} | 削除（デフォルトは 403） |

既存の `/admin/custom-fields` 系は `form_id` パラメータを必須で受けるよう拡張。

### 公開エンドポイント

| メソッド | パス | 変更内容 |
|---------|------|---------|
| GET | /public/form-fields | `?form_id={id}` 引数を追加（省略時デフォルトフォーム） |
| POST | /public/reservations | ペイロードに `form_id` 追加。サーバー側で存在検証 |

## ショートコード

| 記法 | 動作 |
|------|------|
| `[smart_booking]` | デフォルトフォーム |
| `[smart_booking form_id="2"]` | 指定フォーム |
| `[smart_booking store_id="1" form_id="2"]` | 併用可 |
| 存在しない form_id | エラーにせず**デフォルトフォームにフォールバック**（フロントで予約が止まる事故を防止） |

## 削除仕様

| 項目 | 仕様 |
|------|------|
| デフォルトフォーム | **削除不可**（is_system フィールドと同じ思想。「最後の1つを消したら動かない」を構造的に排除） |
| 通常フォーム削除時 | そのフォームの custom_fields も削除（定義はフォームに従属） |
| 予約データ | **reservations / reservation_meta は残す**（担当者削除時と同じポリシー）。予約一覧では「（削除済みフォーム）」表示 |
| 削除確認モーダル | 「このフォームのフィールド定義は削除されます。過去の予約データは残ります」と明記 |

## 予約一覧・CSV への反映

| 項目 | 仕様 |
|------|------|
| 予約一覧 | 「フォーム」列を追加、フィルタにフォーム選択を追加 |
| CSV | 「フォーム名」列を追加（店舗名の隣あたり） |
| 列の表示 | フォームが1つでも**常に表示**（条件で列が増減すると CSV 取り込み側の Excel 運用が壊れるため） |

## 完了条件

- フォームの作成（上限10）・名前変更・削除（デフォルト不可）が動作する
- フォーム追加時に初期フィールド3つが自動生成される
- セレクタ切替でフィールド一覧が正しく切り替わる
- ショートコード `form_id` 指定で該当フォームが表示され、不正 id はデフォルトにフォールバックする
- 予約に form_id が記録され、一覧フィルタ・CSV に反映される
- フォーム削除後も過去予約が「（削除済みフォーム）」として閲覧できる
- 既存環境（v0.3.x）からのマイグレーションが冪等に完了し、既存フィールド・予約がデフォルトフォームに紐づく
- 全フォームが同一スケジュール（空き枠）を共有し、競合防止が従来通り動作する

# 不採用・保留事項（判断記録）

| 項目 | 判定 | 理由 |
|------|------|------|
| 呼び方の管理画面への反映 | ❌ 不採用 | サポート・ドキュメントとの表記乖離が受託モデルに実害 |
| 条件の AND/OR・複数条件 | ❌ 不採用 | 組み合わせ爆発によるバグ温床化を回避 |
| 条件のネスト（2段以上） | ❌ 不採用 | 同上 |
| text 等を条件の親にする | ❌ 不採用 | 部分一致・空文字の仕様が濁る |
| 住所の都道府県/市区町村分割 | ❌ 不採用 | 条件フィールド・CSV との組み合わせ複雑化 |
| 郵便番号データの同梱 | ❌ 不採用 | 約12MB + 更新負担 |
| フォームごとのテーマカラー | ❌ 不採用 | 需要が薄く、設定構造の移行コストに見合わない |
| フォーム複製機能 | 🔵 保留 | 要望が来てから検討 |
| フォーム上限の設定画面公開 | ❌ 不採用 | YAGNI。必要になれば定数変更 or カスタマイズ案件 |
| フォームごとの所要時間・締切 | 📋 カスタマイズ案件 | 本体には入れない（受託の入口として扱う） |
